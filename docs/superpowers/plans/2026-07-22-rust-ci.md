# Rust CI (GitHub Actions) 第1弾 実装計画

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** mac (arm64) / windows の Rust フルビルド+ユニットテスト+アーティファクト添付を GitHub Actions で毎回実行し、バイナリの鮮度とビルド可能性を機械的に保証する。

**Architecture:** 単一ワークフロー `rust-build.yml` に独立並列の 2 ジョブ (build-mac / build-win)。mac は既存 `deploy.sh` を再利用してバンドルまで行い、win は build + 診断 + ベストエフォート収集 (初回実行の診断ログで収集対象を確定する反復前提)。

**Tech Stack:** GitHub Actions (macos-14 / windows-latest) / Rust stable / actions/cache / actions/upload-artifact@v4 / gh CLI (監視)

**Spec:** `docs/superpowers/specs/2026-07-22-rust-ci-design.md`

## Global Constraints

- コミット author は Juha のみ。`Co-Authored-By` を付けない
- 常にフルビルド (PR でも省略しない)。trigger は push(main)/pull_request/workflow_dispatch + paths フィルタ (`cef-unity-rust/**`, ワークフロー自身)
- ユニットテストは `cargo test --workspace --lib --release` (CEF 統合テストと #[ignore] は除外される)
- CEF ダウンロードを引き起こす最初の cargo 実行 (テストステップ) に 1 回リトライ
- target ディレクトリキャッシュは v1 では使わない (~/.cargo のみキャッシュ)
- GitHub ランナーはローカルで再現不能 → 実装後は「push → `gh run watch` で監視 → 修正」の反復を前提とする。**5 回反復しても green にならない場合は停止して方針を再検討する**
- 収集対象が見つからない場合は SKIP をログに残す (silent drop しない)

### Task 1: ワークフロー作成と初回起動

**Files:**
- Create: `.github/workflows/rust-build.yml`

**Interfaces:**
- Produces: artifact 名 `cef-unity-macos-arm64` / `cef-unity-windows-x64` (Task 2/3 が検証)

- [ ] **Step 1: ワークフローを作成**

`.github/workflows/rust-build.yml`:

```yaml
# Rust クライアント/サーバーの CI ビルド (設計: docs/superpowers/specs/2026-07-22-rust-ci-design.md)
# mac は deploy.sh 再利用でバンドルまで、win はビルド+診断+ベストエフォート収集。
name: rust-build

on:
  push:
    branches: [main]
    paths:
      - "cef-unity-rust/**"
      - ".github/workflows/rust-build.yml"
  pull_request:
    paths:
      - "cef-unity-rust/**"
      - ".github/workflows/rust-build.yml"
  workflow_dispatch:

concurrency:
  group: rust-build-${{ github.ref }}
  cancel-in-progress: true

jobs:
  build-mac:
    runs-on: macos-14 # arm64
    steps:
      - uses: actions/checkout@v4

      - uses: dtolnay/rust-toolchain@stable

      - name: Cache cargo registry
        uses: actions/cache@v4
        with:
          path: |
            ~/.cargo/registry
            ~/.cargo/git
          key: cargo-mac-${{ hashFiles('cef-unity-rust/Cargo.lock') }}

      # 初回の cargo 実行は CEF (~1GB) ダウンロードを含むため 1 回リトライ
      - name: Unit tests
        working-directory: cef-unity-rust
        run: cargo test --workspace --lib --release || cargo test --workspace --lib --release

      # 既存スクリプト再利用: release ビルド + server.app バンドル + ad-hoc 署名 +
      # cef-unity-unityproject/Assets/CefUnity/Interop/Plugins/osx-arm64/ へ配置
      - name: Build and bundle (deploy.sh)
        working-directory: cef-unity-rust
        run: ./deploy.sh

      - name: Zip artifact
        working-directory: cef-unity-unityproject/Assets/CefUnity/Interop/Plugins
        run: zip -qry "$GITHUB_WORKSPACE/cef-unity-macos-arm64.zip" osx-arm64

      - uses: actions/upload-artifact@v4
        with:
          name: cef-unity-macos-arm64
          path: cef-unity-macos-arm64.zip

  build-win:
    runs-on: windows-latest
    steps:
      - uses: actions/checkout@v4

      - uses: dtolnay/rust-toolchain@stable

      - name: Cache cargo registry
        uses: actions/cache@v4
        with:
          path: |
            ~/.cargo/registry
            ~/.cargo/git
          key: cargo-win-${{ hashFiles('cef-unity-rust/Cargo.lock') }}

      - name: Unit tests
        working-directory: cef-unity-rust
        shell: bash
        run: cargo test --workspace --lib --release || cargo test --workspace --lib --release

      - name: Build
        working-directory: cef-unity-rust
        shell: bash
        run: cargo build --release

      # Windows のバンドル構成は 2026-05 以来未検証。初回実行はこの診断ログで
      # 収集対象を確定する (設計書に明記済み)。
      - name: Diagnostics
        working-directory: cef-unity-rust
        shell: bash
        run: |
          echo "=== target/release の exe/dll ==="
          ls -la target/release | grep -Ei '\.(exe|dll)$' || echo "(なし)"
          echo "=== cef-dll-sys out ==="
          ls -d target/release/build/cef-dll-sys-*/out/* 2>/dev/null || echo "(なし)"
          for d in target/release/build/cef-dll-sys-*/out/cef_windows_*; do
            [ -d "$d" ] || continue
            echo "--- $d ---"
            ls -la "$d" | head -50
          done

      - name: Collect bundle
        working-directory: cef-unity-rust
        shell: bash
        run: |
          mkdir -p bundle-win
          for f in target/release/cef_unity_rust.dll \
                   target/release/cef-unity-server.exe \
                   target/release/cef-unity-rust-helper.exe; do
            if [ -f "$f" ]; then
              cp "$f" bundle-win/
            else
              echo "SKIP (not found): $f"
            fi
          done
          CEF_DIR=$(ls -d target/release/build/cef-dll-sys-*/out/cef_windows_* 2>/dev/null | head -1)
          if [ -n "$CEF_DIR" ]; then
            mkdir -p bundle-win/cef
            cp -R "$CEF_DIR"/. bundle-win/cef/
          else
            echo "SKIP (not found): cef_windows_* distribution dir"
          fi
          echo "=== bundle-win ==="
          ls -laR bundle-win | head -80

      - uses: actions/upload-artifact@v4
        with:
          name: cef-unity-windows-x64
          path: cef-unity-rust/bundle-win
```

### Task 3: win ジョブの green 化とバンドル確定

**Files:** 必要に応じて Modify: `.github/workflows/rust-build.yml` (Collect bundle ステップ)

**Interfaces:**
- Consumes: Task 1 の run の build-win 診断ログ

- [ ] **Step 1: win ジョブの結果と診断ログを確認**

```bash
cd /Users/juha/Documents/GitHub/cef-unity
RUN_ID=$(gh run list --workflow=rust-build.yml --limit 1 --json databaseId -q '.[0].databaseId')
gh run view "$RUN_ID" --log 2>/dev/null | grep -A60 "=== target/release の exe/dll ===" | head -80
```

ビルド失敗の場合: `gh run view "$RUN_ID" --log-failed | tail -100` で原因特定 →
コンパイルエラーが Rust コード側 (cfg 漏れ等) なら該当箇所を最小修正 → コミット → push。
**最大 5 反復。Rust コードの大規模改修が必要と判明した場合は停止してユーザーへ報告**
(第1弾のスコープはビルド可能性の検証まで — 大規模修正は別プラン)

- [ ] **Step 2: 診断ログに基づき Collect bundle を確定**

診断ログの実ファイル名 (exe/dll の有無、cef_windows_* の実レイアウト) に合わせて
`Collect bundle` ステップの収集対象を修正し、SKIP が想定内のみになるようにする →
`ci: win バンドル収集対象を実レイアウトに合わせて確定` でコミット → push → 再実行確認

- [ ] **Step 3: アーティファクトを確認 (成功基準 2)**
